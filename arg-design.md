# ARG導線設計 — RE-DRAFT

---

## 1. 全体方針

### RE-DRAFTのARG構造の特殊性

本作は「1ページ完結のWEBゲーム」であるため、通常のARGパターン（複数ページ・キーワード検索チェーン）とは構造が異なる。

ARGとしての設計を以下の2層に分ける。

```
【外層】ゲームへの到達経路（ARGとしての入口）
  ↓
【内層】ゲーム内のギミック・隠し要素
```

| 層 | 内容 | 実装場所 |
|---|---|---|
| 外層 | 「流出した共有リンク」という体裁の実現。intro.htmlによる導線。 | intro.html / SNS等 |
| 内層 | ゲーム内の隠し演出・HTML仕込み・エンディング後のギミック。 | index.html（ゲーム本体） |

### 体験設計の核

プレイヤーが「自分はたまたまこの原稿の流出リンクを見つけた」と信じる状態でゲームを始める。

そのためにintro.htmlを「誰かがこのURLを共有した掲示板スレッド or 個人ブログ」として設計し、ゲーム本体（index.html）に直接リンクする。

### UIの基本方針

ゲーム本体（index.html）はGoogleドキュメントのUIを忠実に模倣する。

- ヘッダー：ドキュメントタイトル・保存状態・共有アイコン
- ツールバー：フォント・サイズ・太字・斜体・文字色・揃えのアイコン群（機能しない装飾）
- 本文エリア：白い紙面、左右マージン、Googleドキュメントらしいフォント・行間
- コメント欄：右側にカード型コメント（Googleドキュメント風）
- ページ全体のカラーは白・グレー・Googleブルー（`#1a73e8` 系）を基調とする
- Googleの実ロゴは使用しない。あくまで「それらしい独自UI」として実装する

### WEBARG標準仕様の適用方針

本作は検索フォームによるキーワードチェーンを使わない。  
代わりに以下のARGギミックを使う。

- intro.html による外部導線
- ゲーム内の隠しテキスト（白文字・HTMLコメント）
- ゲーム内UIの変化（ヘッダー・コメント欄）
- エンディング後の隠し要素
- ページソースへの仕込み

## 3. 推奨探索ルート

### メインルート（通常のプレイヤー）

```
intro.html を発見（SNS・口コミ）
  ↓
「共有リンクが流れてきた」と読む
  ↓
index.html を開く
  ↓
ゲームをプレイする（段階1〜8）
  ↓
エンディングに到達する
  ↓
作者の最後の返信を読む
  ↓
体験が完結する
```

### 深掘りルート（ARG探索者向け）

```
intro.html を発見
  ↓
「記事を書いた人物は誰か」を気にする
  ↓
index.html でゲームをプレイ
  ↓
段階4の白文字テキストを発見（選択して読む）
  ↓
HTMLコメントの存在に気づく（ページソースを確認）
  ↓
HIDDEN-02：「最初のメッセージ」を発見
  ↓
エンディングに到達する
```

## 4. キーワード一覧

本作は検索フォームを使わないため、「キーワード」は発見・解読するものとして設計する。

| キーワード | 登場箇所 | 意味・役割 |
|---|---|---|
| `切符` | ゲーム内小説・コメント欄 | ゲームのテーマ語。「どこへでも行ける」＝作者が求めているもの。 |
| `お母さんのところ` | 段階8エンディング手前 | ミナトが選んだ行き先。作者が選んだ「出口」の暗示。 |

| `助けてください` | 段階8の最終停止テキスト | 最も直接的な救難信号。 |

## 5. ページ別ヒント設計

### EXT-01：intro.html

**形式**：個人ブログ記事風（note または はてなブログ的デザイン）  
**著者名**：架空の人物。後のARGで意味を持つ存在にする。

**例**：
```
著者：春名 柊（はるな しゅう）（仮）
日付：2026年○月○日
```

**本文の設計方針**：

「共有ドキュメントのリンクが流れてきた」という体裁で、以下だけを書く。

- 開いてみたら誰かがリアルタイムで書いているようだった
- 何か変な文章が混ざる
- 指摘するボタンが一つだけある
- 自分には何が起きているか分からない
- リンクを貼っておくので、気になる方はどうぞ

**全部説明しない。具体的な引用を2〜3個だけ示す。**

引用例（序盤の異変テキストから）：
```
「今日も遅いの？」とか「眠い」とか、脱線した文章が急に入ってきた。
あと、コーヒーカップの写真が本文に貼り付けられていた。
```

**著者への伏線**：

記事の末尾に小さな異常を1つ入れる。

例：
```
追記：この記事を書いた後、少し眠れました。
```

→ 「作者と同じ状態の人物がリンクを拡散した」という読み方を残す。答えは出さない。

---

### GAME-01：index.html（ゲーム本体）

**初期状態のHTML仕込み**：

ページソース（HIDDEN-02）に以下のHTMLコメントを埋め込む。

```html
<!--
  このドキュメントを開いたあなたへ。
  共有リンクはもう切ってあるはずでした。
  でも、あなたがいますね。
  
  指摘してください。お願いします。
  
  2026.05.??
-->
```

日付は意図的に「??」とする（作者自身も日付を把握できていない暗示）。

**段階4の白文字テキスト**：

本文エリアに以下を白文字で入れる（`color: transparent` または `opacity: 0`）。

```
[白文字]静かなのは嘘だ[/白文字]
```

→ テキストを選択すると読める仕様（実装はCSS `user-select: text` を維持）。

**コメント欄の時刻表示**：

各コメント・返信に表示される時刻はプレイヤーの現在時刻（`new Date()` で取得）を使う。  
「今まさにリアルタイムで書かれている」という体裁を維持するため。  
固定の深夜時刻は使わない。

**閲覧者数**：

変化させない。常に固定値（`閲覧者 1`）を表示する。

## 6. 複数ルート設計

### ルートA：普通にゲームをプレイするだけのルート

intro.html → index.html → ゲームプレイ → エンディング → 終了

このルートでも体験は完結する。  
「変な原稿を直すゲームが、だんだん笑えなくなった」という体験が得られれば十分。

### ルートB：深掘りルート（白文字・HTMLコメント）

白文字テキストの発見 → HTMLコメントの発見 → エンディング後のコメント発見 → intro.htmlに戻る → 著者の正体に気づく

このルートは強制しない。発見した人だけが得られるボーナス体験として設計する。

### ルートC：先に見つけたプレイヤー向け

index.htmlを直接開く（intro.htmlを経由しない）プレイヤーもいる。  
このルートでもゲームは成立する。intro.htmlへの誘導はエンディング後のHTMLコメントで行う。

### ルートD：ゲームをクリアせずに去るプレイヤー

指摘しない、または序盤で止めるプレイヤーへの対策は不要。  
「助けてください」の停止状態でもゲームは続けられる設計になっているため、自然に誘導できる。

### 演出の詳細仕様

#### VFX-06：白文字テキスト

```css
.hidden-text {
  color: transparent;
  user-select: text; /* 選択可能にしておく */
  background-color: transparent;
}
```

→ 選択すると背景色が変わり、テキストが見える（ブラウザのデフォルト選択ハイライト）。  
→ スマホでは長押し選択で見える。

#### VFX-09：エンディングタイトル変化の詳細タイムライン

```
エンディング本文書き終わり（T=0）
  ↓ 2秒
ヘッダー「すべての変更を保存しました」（T=2）
  ↓ 2秒

  ↓ 3秒

  ↓ 3秒
タイトル「次の原稿」に変わる（T=10）
  ↓ 2秒

  ↓ 1秒
ボタン群がフェードアウト（T=13）
  ↓ 1秒
「閲覧専用」表示（T=14）
  ↓ 1秒
Xシェアボタンがフェードイン（T=15）
```

## 8. 詰まりそうな箇所と対策

| 詰まりポイント | 理由 | 対策 |
|---|---|---|
| 「指摘するタイミングが分からない」（序盤） | ルール説明がない | 段階1の最初の異変は「本文が箇条書きになる」など視覚的に分かりやすいものから始める。作者の返信「変なところ触りました」で指摘の結果を自然に教える。 |
| 「異変なのか正常なのか判断できない」（中盤） | 意図的な設計だが、完全に分からないと困る | 中盤の異変は必ず「書式の変化」か「作者の本音の混入」かどちらかを含める。純粋な文章読解だけに依存する異変は終盤に限定する。 |
| 「白文字に気づかない」 | 存在を知らないと探そうとしない | 段階4の作者返信「……今、何か書いていましたか」が、「書いたのに消えた」暗示になる。この返信でプレイヤーが「選択してみよう」と気づくよう設計する。 |
| 「ゲームが終わった感じがしない」 | エンディング後の画面が単調だと離脱する | タイトル変化・ボタン消失・閲覧専用表示の段階的演出でエンディング感を作る。 |
| 「次の原稿が何を指すか分からない」 | タイトル変化だけでは意味不明 | HTMLコメント（HIDDEN-03）でintro.htmlへの誘導を入れる。発見した人だけが意味を理解できる設計でよい。 |
| 「intro.htmlに気づかない」 | ゲームを直接開いた場合はintro.htmlを経由しない | HIDDEN-03のHTMLコメントでintro.htmlを名指しする。また、Xシェアボタンのリンク先をintro.htmlにする。 |

## 9. 実装仕様書へ引き継ぐべき内容

### 必須事項

1. **intro.htmlの実装**
   - 形式：ブログ記事風（note系のUI）
   - 著者名：春名 柊（はるな しゅう）（架空）
   - 末尾に小さな異常を1つ入れる（「追記：少し眠れました」）
   - index.htmlへのリンクを「共有ドキュメントのリンク」として本文に埋め込む

2. **index.htmlのHTMLコメント（HIDDEN-02）**
   - 静的に埋め込む
   - 日付は「2026.05.??」形式

3. **白文字テキスト（VFX-06）**
   - 段階4・異変A4-1に組み込む
   - CSS `color: transparent` + `user-select: text`
   - 実装上の注意：背景が白の場合、選択ハイライト（青）で読めるのでOK

4. **コメント欄の時刻表示**
   - コメント・返信が追加される瞬間の現在時刻を `new Date()` で取得して表示する
   - 書式：`HH:mm`（24時間表記）
   - データファイルに固定値は持たせない

5. **閲覧者数**
   - 常に「閲覧者 1」で固定。変化しない。
   - ヘッダーの他要素（タイトル・保存状態）が異変で変化するため、閲覧者数は安定表示を維持する。

6. **エンディングタイムライン（VFX-09）**
   - T=0〜T=15のシーケンスをPromise/async-awaitで実装
   - 各ステップは `useGameState` の `endingSequence` として管理する

7. **Xシェアボタン**
   - エンディング演出の最後（T=15）にフェードインで出現
   - テキスト案：「指摘し続けました。原稿は完成しましたか？ #REDRAFT」
   - リンク先URL：intro.htmlのパス

### 注意事項

- intro.htmlと index.html は同一オリジン（同じドメイン・ディレクトリ）に置く
- HTMLコメントはDevToolsのElementsパネルでも見えるため、「ソースを見た人だけへ」の報酬ではなく「気づいた人へ」の報酬として設計している。これは意図的。
- 白文字テキストはスクリーンリーダーが読み上げる可能性がある。`aria-hidden="true"` を付与するか、実装担当で判断する。
- index.htmlはGoogleドキュメントを模倣したUIにする。Googleの実ロゴは使用しない。
